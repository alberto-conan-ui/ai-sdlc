# v0.21 — Feedback-Driven Release

**Date:** 2026-03-19

**Summary:** Architectural extensions and refinements driven by real-world feedback from the first v0.2 project (GolemUI). Adds interaction modes, elevates the index to the core navigation primitive, introduces journal folders with handover for session continuity, adds the step node type, and unifies the status vocabulary. The overarching theme is session continuity: every session closes with a clean handover, every session opens by consuming it.

**Status:** Released. Superseded by v0.22.

---

## Changes

### Interaction modes

Planning and Executing are now explicit interaction modes — a top-level process concept orthogonal to stances and workflow stages. In Planning mode, artifacts are provisional (exploring, shaping). In Executing mode, artifacts are authoritative (the design is approved, changes are real). The transition from Planning to Executing is always an explicit human decision. Modes are recorded in `status.md` and referenced throughout the process.

**Files changed:** `process/principles.md` (new Interaction Modes section), `process/workflow.md` (active stack entries show mode), `process/conventions.md` (status vocabulary includes mode).

### Index as navigation primitive

The index file (`[name].index.md`) is elevated from a mandatory authority file to the core navigation mechanism. Every folder in the memory system has one. The grammar is three sections: References (dependency tables), Siblings (peer files in the same folder), Children (child nodes). This applies uniformly to AT, KT, and journal folders.

**Files changed:** `process/memory.md` (index section rewritten), `process/action-tree.md` (index role clarified), `process/knowledge-tree.md` (index role clarified).

### Step node type

Steps are structural containers without gatekeeps. They break a parent into manageable chunks without introducing their own completion criteria — only the parent's gatekeep matters. This fills the gap between phases (gatekept execution containers) and tasks (ungatkept leaves). Steps can hold steps or tasks but not goals, topics, or phases.

Folder naming: `NN.step.name/` (e.g., `05.step.interaction-modes/`).

**Files changed:** `process/action-tree.md` (new Step section, containment rule updated), `process/memory.md` (naming examples), `process/conventions.md` (naming conventions).

### Journal folders with handover

Journal entries evolve from flat files to folders. Each session produces a folder in `journal/live/` containing an index, entry files, and a `handover.md`. The handover is the session continuity mechanism — it captures where work was left, what the next session needs to know, and the Human Lead's stated status assessment.

**Files changed:** `process/journaling.md` (rewritten: folder structure, handover format, session close protocol), `process/conventions.md` (journal naming updated).

### Session continuity protocols

Session open and close are explicitly defined in `roles/operating-rules.md`. Session open reads `status.md`, follows the handover chain, loads the active action path. Session close requires asking the Human Lead for status — the AI does not infer completion. The handover captures the stated assessment.

The Developer exemption (Developer didn't need to follow full session protocol) is removed — all stances follow the same protocol.

**Files changed:** `roles/operating-rules.md` (session protocols), `roles/developer.md` (exemption removed), `bootstrap/ai_readme.template.md` (session start section).

### Status redesign

The status vocabulary is unified into one set for all node types, replacing three separate lists (Action, Phase, Task):

- **Pending** — defined, not started
- **Active** — work in progress (mode says Planning or Executing)
- **Paused** — stopped temporarily
- **Review** — work complete, awaiting Human Lead assessment
- **Done** — Human Lead confirmed (ungatkept nodes: steps, tasks)
- **Achieved** — Human Lead confirmed, gatekeep passed (gatekept nodes: goals, topics, phases)

No action transitions to Done or Achieved without an explicit Human Lead decision.

**Files changed:** `process/conventions.md` (single vocabulary replaces three), `process/action-tree.md` (status references updated).

### Node type property matrix

Node types are reframed as combinations of two properties — container/leaf and gatekept/ungatkept — rather than standalone concepts. A third distinction (gatekeep nature) separates strategic gatekeeps (goals, topics: real-world outcomes, human-judged) from technical gatekeeps (phases: implementation correctness, mechanically verifiable). Stance-based ownership framing removed.

**Files changed:** `process/action-tree.md` (property matrix at top, all types redescribed).

### Terminology decoupling

The workflow stage previously called "Planning" is renamed to "Design." The workflow becomes: Design → Implementation → Review. This ensures "planning" has exactly one meaning in the process: the Planning interaction mode. Verified across all process files.

**Files changed:** `process/workflow.md`, `roles/architect.md`, `process/principles.md` (disambiguation removed).

### Relaxed task workflow

Inside a task, the full stance pipeline doesn't apply. Any stance can do the whole job — the Architect can write code, the Tech Lead can make a design call. Interaction modes still apply. Workflow stages and stance handoffs don't. Tasks are the lightweight escape valve for small changes.

**Files changed:** `process/action-tree.md` (task section), `process/workflow.md`.

### Handover must-ask protocol

Session close requires the AI to ask the Human Lead for the current status rather than inferring it. The handover captures the stated assessment. Closing a session is a logistics event, not a status declaration.

**Files changed:** `process/journaling.md` (session close protocol), `roles/operating-rules.md`.

### Supporting changes

- `roles/common.md` — slimmed to a pointer with three unique duties (Status/AT Root Index, Orientation, External Change Awareness). Summarised content that duplicated `journaling.md` and `operating-rules.md` removed.
- `process/principles.md` — "Strong Reasoning Model" and "The Memory Model" reframed as Prerequisites, not principles. Five genuine principles identified: Simplicity, Human Accountability, Formalise the Implicit, Interaction Modes, Append-Forward.
- `process/memory.md` — goal and step folder examples added to naming section.
- `bootstrap/ai_readme.template.md` — updated to reflect all v0.21 conventions.
- `README.md` — version bumped to v0.21, summary updated.

## Migration Steps (v0.2 → v0.21)

### Structural changes

1. **Journal folders.** Convert `journal/live/` from flat files to folders. Each session gets a folder `YYYY-MM-DD_NN/` containing an index, entry files, and `handover.md`. Existing flat journal files can remain as-is (they predate the convention) or be wrapped in folders if you want consistency.

2. **Status.md format.** Update `status.md` to include the Mode field (Planning, Executing, or —). Add the "Relevant journal" field pointing to the most recent session's handover. Update the active action format to use breadcrumb style.

3. **Status vocabulary.** Replace any per-type status terms with the unified vocabulary: Pending, Active, Paused, Review, Done, Achieved. "Complete" → "Done" (ungatkept) or "Achieved" (gatekept). "Planning" as a status → "Active" (in Planning mode). "Implementing" → "Active" (in Executing mode).

### Behavioral changes

4. **Session close protocol.** The AI must ask the Human Lead for status at session close, not infer it. Update your session habits accordingly.

5. **Workflow stage language.** Use "Design" instead of "Planning" when referring to the workflow stage. "Planning" now means only the interaction mode.

6. **Task workflow.** Tasks no longer require full stance ceremony. Any stance can handle the full task lifecycle.

### No structural migration needed

The following changes take effect automatically when the methodology repo is updated:

- Node type property matrix (documentation change, not file structure)
- Relaxed task workflow (behavioral, not structural)
- `common.md` restructuring (methodology file, not project memory)
- `principles.md` reorganization (methodology file, not project memory)
