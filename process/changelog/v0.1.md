# v0.1 — Initial Process (baseline)

**Date:** Pre-2026-03-17 (documented retroactively)

**Summary:** The original AI-SDLC methodology as designed and first deployed. This version was used on the GolemUI project across 16 sessions. It established the core concepts — action tree, knowledge tree, role-based stances, gatekeeping, and append-forward — but included ceremony that real-world usage revealed as excessive or unused.

---

## Memory Model

- **Two trees + journal.** The action tree (short-term memory) and knowledge tree (long-term memory) are the primary structures. The journal sits between them as a bridge and audit trail.
- **Action tree files:** `gatekeep.md` (required), `context.md`, `index.md`, `log.md`, `KEY_INSIGHTS.md`, `phases/` with phase specs, prompt plans, implementation prompts, and session receipts.
- **Knowledge tree files:** `index.spec.md` at each node, `*.spec.md` leaf files for deep knowledge. Organised by codebase boundaries.
- **Journal:** Weekly files named `YYYY-WNN.md`. Entry types: `[decision]`, `[observation]`, `[process-change]`. Cross-cutting only — action-scoped events belong in `log.md`.
- **Memory flow:** Sessions produce log entries and KEY_INSIGHTS entries in the action tree. Cross-cutting observations go to the journal. On action completion, KEY_INSIGHTS are reviewed and migrated to the knowledge tree. The Curator audits the pipeline using the journal as input.
- **Archiving:** Single root-level `archive/` folder alongside `action-tree/`. Completed action subtrees move there. No archive mechanism for the knowledge tree or journal.

## Roles

Six roles plus the Human Lead:

- **Architect** — designs approach, writes phase specs, challenges assumptions, pushes back on scope. Writes directly to knowledge tree. Entry point: `roles/architect.md`.
- **Tech Lead** — translates specs into implementation prompts. Writes to `KEY_INSIGHTS.md` and `log.md`. Entry point: `roles/tech-lead.md`.
- **Developer** — executes prompts literally, produces session receipts. Loads only the prompt. Entry point: `roles/developer.md`.
- **Navigator** — advises on next steps, produces briefings and handoff prompts. Reads STATUS.md and journal. Entry point: `roles/navigator.md`.
- **Curator** — audits the memory pipeline. Reads journal, logs, KEY_INSIGHTS, knowledge tree. Proposes editorial actions. Entry point: `roles/curator.md`.
- **Bootstrapper** — one-time setup of `.ai-sdlc/` folder structure. Entry point: `roles/bootstrapper.md`.

Role boundaries are expressed as prohibitions ("don't write code", "don't redesign the spec").

## Role Infrastructure

- `roles/operating-rules.md` — five operating rules loaded by every AI role: identify yourself, the human is the authority, explain before you act, append-forward, stay in your lane.
- `roles/common.md` — shared responsibilities: memory and recording, process anchoring, STATUS.md updates, orientation.

## Process Documents

- `process/principles.md` — three core principles: human accountability, formalise the implicit, append-forward. Plus a pointer to the memory model.
- `process/memory.md` — defines the two trees, the journal, how memory flows, the KEY_INSIGHTS lifecycle, and recording responsibilities by role.
- `process/journaling.md` — the recording system: pipelines by role (design vs implementation), the "who writes what" matrix, session recording checklists (per-role and on action completion), the append-forward rule, and recording anti-patterns.
- `process/workflow.md` — the open flow, the active stack, the planning-implementation loop, phase handovers, action completion, staying current, session persistence.
- `process/roles.md` — the role overview: Human Lead, core flow (Architect -> Tech Lead -> Developer), situational stances (Navigator, Curator), one-time (Bootstrapper), common foundation, model tiers.
- `process/action-tree.md` — tree structure, node files, gatekeeping model, active stack, recording, growth, completion, archiving.
- `process/knowledge-tree.md` — tree structure, index files, leaf files, growth, monorepo patterns, knowledge file format, what belongs at each level, maintenance.
- `process/conventions.md` — naming conventions, status vocabularies (action, phase, receipt), bootstrapper templates.
- `process/prompts.md` — prompt craft principles and sequencing.
- `process/anti-patterns.md` — common failure modes.

## Conventions

- Project memory folder: `.ai-sdlc/memory/` (gitignored inside the code repo)
- Action folders: `kebab-case-name/`
- Phase folders: `N-kebab-case-name/`
- Implementation prompts: `NN-kebab-case-description.md`
- Journal files: `YYYY-WNN.md`
- Action statuses: Active, Achieved, Paused, Pending
- Phase statuses: Complete, Implementing, Planning, Not planned yet, Paused
- Receipt statuses: Complete, Partial, Blocked

## Model Tiers

Two tiers:

- **Tier 1 — Reasoning:** Architect, Tech Lead (complex phases), Curator
- **Tier 2 — Execution:** Developer, Tech Lead (straightforward phases), Navigator

## Known Issues (from GolemUI usage, 16 sessions)

These observations motivated the v0.2 redesign. They are documented here as part of the v0.1 record, not as a to-do list:

- The handoff pipeline (Architect -> Tech Lead -> Developer in separate sessions) was never used. Roles were combined in all 16 sessions.
- The Tech Lead role was never invoked independently. Architect specs were detailed enough for direct implementation.
- The Curator role was never invoked. KT audit responsibilities were performed informally by the Architect.
- Journal entries, KEY_INSIGHTS.md files, and session receipts were never produced. `log.md` captured everything.
- `context.md` duplicated KT content and went stale. It worked better as a pointer to knowledge tree nodes.
- No mechanism to classify work complexity (pattern-following vs novel design).
- Phase specs had no versioning discipline.
- Gatekeeps repeated boilerplate universal checks.
- The methodology didn't state its model requirements (tier-3 assumed but never said).
- KT restructuring had no protocol for stale cross-references.

## Migration Steps

**This is the baseline.** No migration steps — this version describes the process as originally shipped. Projects created under v0.1 have this structure by default.
