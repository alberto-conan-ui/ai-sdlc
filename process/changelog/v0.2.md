# v0.2 — Process Redesign

**Date:** 2026-03-17 (development started)

**Summary:** Redesign based on nine design principles developed from real-world usage evidence. Simplifies the process, fixes structural issues in the memory model, reframes roles as cognitive stances, adds the Auditor role, introduces process versioning. Development tracked in the `process-redesign` action.

**Status:** Released. See `process-redesign/context.md` for the full design principles and execution history.

---

## Changes (accumulated per phase)

### Phase 1 (Auditor + version snapshot)

- **Added:** `roles/auditor.md` — new role for process health evaluation and version migration.
- **Added:** `process/changelog/` — version changelog folder with per-version files and migration steps.
- **Added:** v0.1 formally documented as baseline in `process/changelog/v0.1.md`.
- **Changed:** Archiving convention. Each memory layer now has its own `archive/` subfolder instead of a single root-level archive. Action tree: `action-tree/archive/` (moved from root-level `archive/`). Knowledge tree: `knowledge-tree/archive/` (new — for retired insights and superseded structures). Journal: `journal/archive/` (new — for processed entries). This makes archiving consistent across all three layers and supports version migration (old structures move to the relevant tree's archive, not a shared dump).

### Phase 2 (Principles)

- **Changed:** `process/principles.md` rewritten. Simplicity added as the first principle (was entirely absent in v0.1). Tier-3 model requirement added as a new principle (was implicit in v0.1, never stated). ### Phase 3 (Memory model)

- **Changed:** `process/memory.md` rewritten. "Two trees + journal bridge" replaced with "three equal layers" — journal elevated from bridge/audit-trail to first-class temporal intake buffer. - **Changed:** `process/journaling.md` rewritten. Simplified recording system — journal is now the primary recording destination for all sessions (was only for cross-cutting observations in v0.1). Curator role removed from recording matrices (role eliminated in principle, formal removal in Phase 4). Auditor added to design roles. Session checklists simplified. - **Changed:** `process/changelog/index.md` — reference header converted to labeled table format.

## Migration Steps (v0.1 -> v0.2)

*Accumulated from per-phase migration notes. Each step cites the phase that introduced the change.*

### Phase 1 migration impact

1. **Move root-level archive into action tree.** Rename `archive/` to `action-tree/archive/`. If the project has completed actions in the old root-level archive, move them. Update any references in STATUS.md that point to `../archive/` to point to `./archive/` (now relative within action-tree).
2. **Create knowledge tree archive.** Create `knowledge-tree/archive/`. No content to move yet — this is infrastructure for future retirements and migrations.
3. **Create journal archive.** Create `journal/archive/` (and `journal/live/` if adopting the v0.2 journal structure). Move any existing processed journal entries to `journal/archive/`.
4. **New files.** Copy `roles/auditor.md` and the `process/changelog/` folder into the project's methodology reference (or update the methodology repo the project points to).

### Phase 2 migration impact

No structural changes to existing projects. The principles file is methodology documentation, not project memory. Projects that reference `process/principles.md` will pick up the new content automatically when the methodology repo is updated.

### Phase 3 migration impact

1. **Create journal folder structure.** Create `journal/live/` and `journal/archive/` if they don't exist. If the project has existing journal files (v0.1 weekly files `YYYY-WNN.md`), decide per file: unprocessed entries move to `journal/live/`, already-processed entries move to `journal/archive/`.
2. **Journal naming convention change.** v0.1 used weekly files (`YYYY-WNN.md`). v0.2 uses per-session files (`YYYY-MM-DD_NN.md`). Existing files can remain as-is in their archive location — the naming convention applies to new entries going forward.
3. **Existing `log.md` entries.** No change required. Action-scoped `log.md` files remain valid and serve the same purpose in v0.2 (multi-session continuity on a single action). The change is that the journal now captures the session narrative that v0.1 expected `log.md` to carry alone.
4. **Add reference headers.** Existing project memory files (knowledge tree specs, action context files) should gain reference header tables as they are touched in future sessions. This is not a bulk migration — apply incrementally as files are edited.

### Phase 4 migration impact

No structural changes to existing project memory. The role changes are behavioural, not file-level — they affect how AI stances operate, not what files exist in the project.

1. **Role invocation language.** Sessions should use "stance" instead of "role" when referring to AI cognitive modes. This is a terminology shift, not a file change. Existing project files that reference "roles" (e.g., journal entries, log entries) remain valid and don't need updating.
2. **Curator sessions.** Projects that previously ran Curator sessions should shift that responsibility to the Architect: periodic KT health checks during normal design work, KEY_INSIGHTS review on action completion, and journal processing on demand. No special migration step — the Architect picks up these responsibilities naturally.
3. **Adaptive flow gate.** Projects previously using the fixed Architect → Tech Lead → Developer pipeline can continue doing so for complex work — the adaptive flow gate doesn't prohibit the full pipeline, it makes the Tech Lead situational rather than mandatory. No migration step required.
4. **Bootstrapper.** Projects already bootstrapped are unaffected. The Bootstrapper was a one-time role by definition. New projects will follow updated bootstrap guidance (to be finalised in Phase 5).

### Phase 4 refinement migration impact

No additional structural changes to existing project memory beyond Phase 4.

1. **Navigator sessions.** Projects that previously used the Navigator should shift to using "where are we?" in any stance (already a common responsibility) and the Architect for handoff prompts when opening fresh sessions. No special migration step — the Navigator's responsibilities are already covered.
2. **Adaptive flow gate actor.** Projects previously assessing phases with the Architect should shift to invoking the Tech Lead at the gate. This is a behavioural change in session practice, not a file-level migration.

### Phase 5 migration impact

No structural changes to existing project memory. Phase 5 changes are methodology-level (process docs, role files, bootstrap guides) and take effect when the project updates its methodology reference.

1. **Dead role files.** If the project's methodology reference is a clone (not a symlink), delete `roles/curator.md`, `roles/navigator.md`, `roles/bootstrapper.md` from the local copy. If it's a symlink, the deletion propagates automatically on pull.
2. **Bootstrap guidance.** Projects already bootstrapped are unaffected. New projects should follow the updated `bootstrap/README.md` § Step B.
3. **Journal file type.** The journal-entry file type definition changed significantly. Projects producing journal entries should follow the v0.2 format (per-session files, unstructured narrative) going forward. Existing v0.1 weekly journal files remain valid in their archive location.
4. **Hierarchy discipline.** The new "Building Good Hierarchies" guidance in `memory.md` applies to all future tree modifications. Existing tree structures don't need retroactive restructuring — apply the guidance when the tree is next modified.
