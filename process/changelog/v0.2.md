# v0.2 — Process Redesign (in progress)

**Date:** 2026-03-17 (development started)

**Summary:** Redesign based on nine design principles developed from real-world usage evidence. Simplifies the process, fixes structural issues in the memory model, reframes roles as cognitive stances, adds the Auditor role, introduces process versioning. Development tracked in the `process-redesign` action.

**Status:** In progress. Changes are documented per phase as they land. See `process-redesign/context.md` for the full design principles and execution plan.

---

## Changes (accumulated per phase)

### Phase 1 (Auditor + version snapshot)

- **Added:** `roles/auditor.md` — new role for process health evaluation and version migration.
- **Added:** `process/changelog/` — version changelog folder with per-version files and migration steps.
- **Added:** v0.1 formally documented as baseline in `process/changelog/v0.1.md`.
- **Changed:** Archiving convention. Each memory layer now has its own `archive/` subfolder instead of a single root-level archive. Action tree: `action-tree/archive/` (moved from root-level `archive/`). Knowledge tree: `knowledge-tree/archive/` (new — for retired insights and superseded structures). Journal: `journal/archive/` (new — for processed entries). This makes archiving consistent across all three layers and supports version migration (old structures move to the relevant tree's archive, not a shared dump).

### Phase 2 (Principles)

- **Changed:** `process/principles.md` rewritten. Simplicity added as the first principle (was entirely absent in v0.1). Tier-3 model requirement added as a new principle (was implicit in v0.1, never stated). The Memory Model pointer updated to reference "three layers" instead of "two trees" and removed the Curator-specific audit reference. Human Accountability, Formalise the Implicit, and Append-Forward unchanged in substance.

### Phase 3 (Memory model)

- **Changed:** `process/memory.md` rewritten. "Two trees + journal bridge" replaced with "three equal layers" — journal elevated from bridge/audit-trail to first-class temporal intake buffer. Journal structure defined: `journal/live/` and `journal/archive/`. Per-tree archiving documented for all three layers. New structural conventions section added: reference headers (labeled, ordered dependency tables), one-file-per-folder rule (KT and AT only), hierarchy as discipline.
- **Changed:** `process/journaling.md` rewritten. Simplified recording system — journal is now the primary recording destination for all sessions (was only for cross-cutting observations in v0.1). Curator role removed from recording matrices (role eliminated in principle, formal removal in Phase 4). Auditor added to design roles. Session checklists simplified. Journal-specific anti-pattern added (skipping journal processing). KEY_INSIGHTS retained for Tech Lead but simplified treatment.
- **Changed:** `process/principles.md` — reference header converted from inline blockquote to labeled table format. Content unchanged from Phase 2.
- **Changed:** `process/changelog/index.md` — reference header converted to labeled table format.

*Subsequent phases will append their changes here as they land.*

## Migration Steps (v0.1 -> v0.2)

*Will be finalised in Phase 6 after all changes are complete and tested. Per-phase migration impact is documented in `process-redesign/context.md` and accumulated here as phases complete.*

### Phase 1 migration impact

1. **Move root-level archive into action tree.** Rename `archive/` to `action-tree/archive/`. If the project has completed actions in the old root-level archive, move them. Update any references in STATUS.md that point to `../archive/` to point to `./archive/` (now relative within action-tree).
2. **Create knowledge tree archive.** Create `knowledge-tree/archive/`. No content to move yet — this is infrastructure for future retirements and migrations.
3. **Create journal archive.** Create `journal/archive/` (and `journal/live/` if adopting the v0.2 journal structure). Move any existing processed journal entries to `journal/archive/`.
4. **New files.** Copy `roles/auditor.md` and the `process/changelog/` folder into the project's methodology reference (or update the methodology repo the project points to).

### Phase 2 migration impact

No structural changes to existing projects. The principles file is methodology documentation, not project memory. Projects that reference `process/principles.md` will pick up the new content automatically when the methodology repo is updated.

### Phase 3 migration impact

1. **Create journal folder structure.** Create `journal/live/` and `journal/archive/` if they don't exist. If the project has existing journal files (v0.1 weekly files `YYYY-WNN.md`), decide per file: unprocessed entries move to `journal/live/`, already-processed entries move to `journal/archive/`.
2. **Journal naming convention change.** v0.1 used weekly files (`YYYY-WNN.md`). v0.2 uses per-session files (`YYYY-MM-DD_NN.md`). Existing files can remain as-is in their archive location — the naming convention applies to new entries going forward.
3. **Existing `log.md` entries.** No change required. Action-scoped `log.md` files remain valid and serve the same purpose in v0.2 (multi-session continuity on a single action). The change is that the journal now captures the session narrative that v0.1 expected `log.md` to carry alone.
4. **Add reference headers.** Existing project memory files (knowledge tree specs, action context files) should gain reference header tables as they are touched in future sessions. This is not a bulk migration — apply incrementally as files are edited.
